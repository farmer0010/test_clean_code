# 🚀 Day 1: 클린 코드와 단위 테스트(Unit Test)의 기초

## 1. 우리의 핵심 철학
> **"단순히 예쁜 코드가 아니라, 상품 관점에서 가치 있는 클린 코드"**
코드를 예술 작품처럼 꾸미는 것이 아니라, 앱이 뻗지 않게 막고 유지보수 비용을 낮춰 비즈니스를 지키는 것이 진짜 클린 코드다.

## 2. 프로젝트 디렉터리 분리
실제 서비스되는 코드와 테스트 코드는 반드시 물리적으로 분리되어야 한다.
- `src/main/java`: 진짜 요리를 하는 본점 영업장 (프로덕션 코드)
- `src/test/java`: 상품을 망치로 때려보는 품질 검증 연구소 (테스트 코드)
- **이유:** 배포 시 테스트 코드가 섞여 나가면 용량이 커지고 보안 위험이 생기기 때문.

## 3. 단위 테스트 (Unit Test)란?
- 프로그램의 가장 작은 단위(메서드)가 정상 작동하는지 확인하는 자동화된 보험. 

[Image of Software Testing Pyramid]

- 수백만 명이 쓰는 서비스에서 내 코드 한 줄 때문에 전체 시스템이 멈추는 것을 막아준다.

## 4. 테스트 케이스 설계의 3단계 (생각의 순서)
1. **Happy Path (정상 흐름):** 우리가 의도한 대로 유저가 정상적인 값을 넣었을 때. (가장 중요)
2. **Boundary Value (경계값):** 기준이 되는 바로 그 턱걸이 값. 개발자가 가장 실수를 많이 하는 구간. (예: 8자리 이상일 때, 딱 8자리 입력)
3. **Edge Case (예외/극단적 상황):** 해커나 유저의 실수로 들어오는 비정상적인 값. (예: `null` 또는 빈 문자열 `""`)

## 5. 클린 코드 원칙: Guard Clause (보호 구문)
- 본격적인 로직을 실행하기 전에, 메서드 최상단에서 불량 데이터(`null` 등)를 방어막으로 튕겨내는 기법.
- **Fail-Fast (빨리 실패하기):** 에러가 날 거라면 시스템 깊숙한 곳에서 `NullPointerException`으로 뻗게 두지 말고, 입구에서 `IllegalArgumentException`으로 통제 가능하게 예쁘게 포장해서 던져야 한다.

## 6. 필수 JUnit 5 테스트 도구
- `@Test`: 이 메서드가 테스트 코드임을 인텔리제이에 알림.
- `assertDoesNotThrow()`: 에러가 터지지 않아야 성공! (Happy Path 검증용)
- `assertThrows(에러.class)`: 반드시 내가 지정한 에러가 터져야 성공! (Edge Case 검증용)
